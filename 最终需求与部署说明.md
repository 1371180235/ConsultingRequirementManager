# 咨询项目全流程需求管理系统最终需求与部署说明

文档版本：v1.0  
功能基线：2026-07-10  
适用代码：`本地/` SQLite 版、`远程/` MySQL 版

## 1. 建设目标

系统面向咨询项目负责人，以资金规划为主轴、落地版本为隔离边界、需求任务为最小管理颗粒度，贯通宏观规划、年度拆分、建设落地、招投标、交付验收和运维运营。系统需要解决多来源需求散落、跨角色同步重复、版本内容混淆、资金去向不清和需求响应慢的问题。

核心业务链路：

`规划项目 -> 年度计划 -> 落地版本 -> 需求任务 -> 资金流水/成果物 -> 验收与运维`

## 2. 使用角色与边界

| 角色 | 核心关注内容 | 主要操作 |
| --- | --- | --- |
| 管理员 | 用户、权限、全量数据、审计和备份 | 用户管理、密码重置、审批、备份恢复 |
| 咨询负责人 | 项目规划、版本、需求、资金和跨角色协同 | 创建项目/年度/版本、需求管理、审批变更 |
| 客户 | 自身需求、处理进度、版本规划 | 提交需求、查看进度和反馈 |
| 销售 | 项目进展、资金申报、投入汇总 | 提交需求、查看资金、导出进展材料 |
| 项目经理 | 交付进度、验收、投入和风险 | 编辑需求、状态流转、资金登记、成果物挂载 |
| 研发人员 | 版本需求池、优先级、研发状态 | 编辑任务、状态流转、上传交付材料 |
| 运营人员 | 上线需求、线上问题、运营反馈 | 提交/编辑运维需求、状态流转、上传反馈 |

远程版角色由登录账号决定。客户账号必须由管理员授权具体项目，项目下拉、项目列表、首页项目数和全局搜索只返回已授权项目。客户、研发人员和运营人员不显示项目预算、需求资金和资金导出。系统设置仅管理员和咨询负责人可见，用户管理和附件恢复仅管理员可执行。

本地版的角色切换用于单机演示和工作视角预览，不构成身份认证或安全边界。包含敏感数据的正式多人场景必须使用远程版。

## 3. 业务阶段与成果物

| 阶段 | 核心工作 | 成果物 | 挂载对象 |
| --- | --- | --- | --- |
| 1. 宏观规划 | 明确项目整体方向与资金盘 | 可研报告 | 规划项目 |
| 2. 规划细化 | 拆分年度任务、痛点、功能和预算 | 分年任务申报书 | 年度计划 |
| 3. 建设落地 | 细化功能点、执行要求和交付任务 | 任务书方案、需求清单 | 落地版本 |
| 4. 招投标 | 明确标的范围和招应标要求 | 招标文件、应标文件 | 落地版本 |
| 5. 交付验收 | 验证版本交付并归档总结 | 验收报告、项目总结 | 落地版本 |
| 6. 运维运营 | 上线问题处理、推广和持续运营 | 运维反馈、运营反馈 | 需求任务 |

当版本需求全部进入待验收、已上线运维或已关闭状态且缺少验收材料时，系统提示补充验收报告或项目总结。

## 4. 核心数据模型

| 实体 | 关键关联 | 关键约束 |
| --- | --- | --- |
| 规划项目 | 顶层实体 | 项目编号唯一，总预算不小于 0 |
| 年度计划 | 归属规划项目 | 年度预算不小于 0，且不超过项目总预算 |
| 落地版本 | 归属项目和年度计划 | 版本预算不小于 0，且不超过年度预算 |
| 需求任务 | 归属项目，可进入待规划池或关联版本 | 需求编号唯一，预估预算不小于 0 |
| 资金流水 | 关联项目/年度/版本，可关联需求 | 分配、实际消耗、调整必须关联需求 |
| 成果物 | 挂载项目、年度、版本或需求 | 挂载对象必须存在且属于当前项目 |
| 版本基线 | 关联冻结版本 | 保存冻结时预算汇总和需求核心快照 |
| 变更申请 | 关联冻结版本和需求 | 审批通过后才应用修改或软删除 |
| 状态历史/操作日志 | 关联需求或业务对象 | 记录操作人、时间、前后状态和原因 |
| 用户 | 关联角色和客户项目授权 | 用户名唯一、账号可停用、密码只存哈希 |

## 5. 需求生命周期

新需求可以直接进入指定未冻结版本，也可以进入待规划池。待规划需求确认后分配到当前版本；冻结版本不允许直接接收新需求或移出原有需求。

允许的状态流转如下：

| 当前状态 | 允许的目标状态 |
| --- | --- |
| 草稿 | 规划中、已取消 |
| 规划中 | 已排期、已驳回、已挂起、退回修改 |
| 已排期 | 研发中、已挂起、已取消、退回修改 |
| 研发中 | 待验收、已挂起、退回修改 |
| 待验收 | 已上线运维、退回修改 |
| 已上线运维 | 已关闭、变更中 |
| 已关闭 | 变更中 |
| 已驳回 | 草稿、已取消 |
| 已挂起 | 规划中、已排期、研发中、已取消 |
| 已取消 | 草稿 |
| 变更中 | 规划中、已排期、研发中、待验收 |
| 退回修改 | 规划中、已排期、研发中、已取消 |

每次流转必须填写说明，采用原状态条件更新防止并发覆盖；需求状态、状态历史和操作日志在同一事务提交。进入已关闭时记录实际完成日期，离开已关闭时清理该日期。

## 6. 版本与变更控制

1. 版本冻结时在同一数据库事务内生成基线、写入需求核心快照并更新冻结状态。
2. 基线记录版本预算、需求数量、已分配预算、实际消耗，以及需求编号、名称、状态、优先级和资金快照。
3. 冻结版本中的需求编辑和删除自动形成变更申请，不直接修改业务数据。
4. 管理员或咨询负责人审批；审批状态更新、需求修改/软删除、状态历史和审计日志在同一事务内提交。
5. 跨版本比对按标准化需求名称匹配，展示新增、移除、变更和一致项；有资金权限的角色同时看到预算差异。

## 7. 资金规则

资金支持项目总预算、年度预算、版本预算、需求分配预算和实际消耗四级穿透。

- 金额必须是有限数值，拒绝 `NaN`、正负无穷和非法文本。
- 年度必须在 2000-2100；日期必须为有效的 `YYYY-MM-DD`，版本结束日期不得早于开始日期。
- 项目、年度、版本、需求预估预算不得为负数。
- 已分配预算、实际消耗和调整金额只能通过资金流水修改，不能在需求编辑表单中直接覆盖。
- 除调整金额外，流水金额必须大于 0；调整金额不能为 0，调整后需求分配预算不得小于 0。
- 版本下全部需求的分配预算不得超过版本预算。
- 实际消耗超过需求已分配预算时必须二次确认并显示风险提示。
- 版本冻结后只允许继续登记实际消耗，不允许新增计划、分配、调整或冻结金额流水。
- 远程版使用 MySQL 行锁和事务，避免多客户端同时登记资金导致丢失更新。

## 8. 页面与联动

- 左侧导航、顶部项目/年度/版本上下文和右侧内容区构成统一桌面工作台。
- 项目切换联动年度和版本；年度切换联动版本；版本切换刷新当前页面。
- 首页状态卡片可跳转到对应需求筛选结果。
- 需求列表支持范围、状态筛选、表头排序和双击详情。
- 资金穿透图的需求节点可双击进入需求详情。
- 长页面和长表单支持滚动，弹窗高度受屏幕约束。
- 表格金额按千分位显示，状态、风险和冻结信息使用一致的语义色。
- CSV 导出对公式起始字符进行转义，降低表格公式注入风险。

## 9. 两种运行版本

| 项目 | 本地 SQLite 版 | MySQL 远程版 |
| --- | --- | --- |
| 使用场景 | 单人、离线、演示、轻量项目 | 多人共享、局域网或 VPN 内正式协同 |
| 数据库 | `data/app.db` | Linux MySQL 8.x |
| 登录 | 无，角色仅为视角预览 | 用户名密码登录，PBKDF2-HMAC-SHA256 哈希 |
| 并发 | 单进程为主 | 预算、冻结、审批使用数据库事务和行锁 |
| 附件 | 本机 `data/attachments` | 服务器共享目录或阿里云 OSS 二选一 |
| 备份 | 应用 ZIP 包含数据库和附件 | `mysqldump` 备份数据库；附件独立备份 |

## 10. MySQL 与附件部署模型

远程版是 Tkinter 桌面客户端，Linux 服务器运行 MySQL；Linux 服务器不需要运行 GUI。桌面客户端会直接连接 MySQL，因此应仅在受控内网或 VPN 中使用，禁止把 3306 直接暴露到公网。

附件后端由 `config/mysql_config.json` 的 `attachment_storage` 统一选择：

### 10.1 服务器目录模式

```json
{
  "attachment_storage": "server",
  "attachments_dir": "/srv/consulting-requirement-manager/attachments"
}
```

应用进程必须能直接访问该目录。若客户端运行在 Windows、附件目录位于 Linux 文件服务器，应先通过 SMB/NFS 将目录挂载为所有客户端可访问的共享路径，例如 `\\fileserver\\crm-attachments`。MySQL 服务器上的 `/srv/...` 路径不会自动对 Windows 客户端可见。

### 10.2 阿里云 OSS 模式

```json
{
  "attachment_storage": "oss",
  "oss_endpoint": "https://oss-cn-hangzhou.aliyuncs.com",
  "oss_bucket": "your-private-bucket",
  "oss_prefix": "consulting-requirement-manager"
}
```

OSS 凭据只能通过环境变量提供：`OSS_ACCESS_KEY_ID`、`OSS_ACCESS_KEY_SECRET`，临时凭据可增加 `OSS_SECURITY_TOKEN`。Bucket 应设为私有，账号仅授予指定前缀所需的上传和删除权限，并启用版本控制、生命周期和审计日志。

## 11. 安全与运维要求

- MySQL 密码优先从 `CRM_DB_PASSWORD` 环境变量读取；配置文件中的 `password` 仅用于受控兼容场景。
- 首次启动远程版通过 `CRM_INITIAL_ADMIN_PASSWORD` 初始化管理员密码，或由界面引导设置；初始化后应清理该环境变量。
- MySQL 使用独立最小权限账号、TLS CA 校验、来源 IP 白名单和主机防火墙。
- 生产配置保持 `create_database=false`、`seed_demo_data=false`。
- 管理员可创建、停用账号和重置密码；用户可修改本人密码；禁止停用当前登录账号。
- 客户账号采用显式项目白名单；新建客户在管理员授权前看不到任何项目。
- 数据库、附件和配置分别备份，定期执行恢复演练。
- CSV、备份 ZIP 和日志都可能包含业务敏感信息，应纳入访问控制和留存策略。

建议备份目标：MySQL 每日全量并结合 binlog，RPO 不高于 24 小时；关键项目建议 RPO 1 小时。附件服务器每日增量，OSS 启用版本控制。建议季度恢复演练，明确业务可接受的 RTO。

## 12. 上线验收清单

- [ ] Linux MySQL 版本、字符集、时区、存储空间和备份策略已确认。

- [ ] 应用使用非 root 最小权限账号，密码由环境变量注入。
- [ ] MySQL TLS 已启用，客户端配置 `ssl_ca` 并完成证书校验。
- [ ] `create_database=false`、`seed_demo_data=false`。
- [ ] 首个管理员密码已初始化并完成改密测试。
- [ ] 各角色账号逐一验证菜单、资金字段、写操作和导出权限。
- [ ] 项目、年度、版本、需求、资金、冻结、审批主链路验收通过。
- [ ] 服务器目录或 OSS 上传、异常清理和备份恢复验收通过。
- [ ] 多客户端同时登记预算、冻结版本和审批变更的并发测试通过。
- [ ] 本地版执行 `python selftest.py`；远程版在目标 MySQL 执行 `python selftest.py`。
- [ ] MySQL `mysqldump` 恢复和附件恢复已在非生产环境演练。

## 13. 已知边界与后续演进

当前远程版是桌面客户端直连 MySQL/OSS，适用于单组织可信内网协同，但不适合作为互联网、多租户或零信任架构。客户角色已有项目白名单，内部角色仍按单组织模型查看全部项目；客户端仍需持有数据库访问凭据，应用层角色权限也不能替代数据库隔离。面向公网或跨组织部署时，应增加 HTTPS API 服务，由服务端统一处理鉴权、行级权限、OSS 临时凭据和审计。

当前建表逻辑使用 `CREATE TABLE IF NOT EXISTS`，尚无正式的 schema 版本和迁移框架；正式长期运维前应补充版本化 SQL 迁移。现有表未全面声明外键和数据库级检查约束，主要依靠应用校验。标签仍为文本，多选标签字典、可配置看板、报告模板和稳定的跨版本业务需求主键属于后续增强项。

在完成本章上线验收前，不应将“代码可运行”直接等同于“生产环境已验收”。
