# 远程版 Linux 部署说明

更新日期：2026-07-17
适用范围：咨询项目全流程需求管理系统远程 Web 版
部署目标：完整功能和数据库运行在 Linux 服务器，用户只通过浏览器访问 HTTPS 域名

## 1. 部署完成后的访问地址

假设在 `.env` 中配置：

```text
DOMAIN=crm.example.com
```

部署完成后用户访问：

- 系统入口：`https://crm.example.com/`
- 健康检查：`https://crm.example.com/health`

请把示例域名替换为实际业务域名。浏览器不需要安装客户端，也不持有数据库地址、账号或密码。

服务器端结构：

```text
Internet
  -> TCP 80/443、UDP 443
  -> Caddy（HTTPS 证书、Web 静态文件、反向代理）
  -> 内部 proxy 网络 -> FastAPI 服务端 API
  -> 内部 database 网络 -> MySQL 8.4 LTS
```

禁止向公网开放 MySQL `3306` 和 API 容器 `8000` 端口。

## 2. 服务器与软件基线

推荐生产环境：

| 项目 | 要求 |
| --- | --- |
| Linux | Ubuntu Server 24.04 LTS 64 位（推荐） |
| CPU | 最低 2 核，推荐 4 核 |
| 内存 | 最低 4 GB，推荐 8 GB |
| 磁盘 | 最低 40 GB；按附件和备份规模扩容 |
| Docker Engine | 27.0 或更高稳定版 |
| Docker Compose | v2.29 或更高插件版 |
| 数据库 | Oracle MySQL Community Server 8.4 LTS 容器 |
| 反向代理 | Caddy 2.10 系列容器 |
| 浏览器 | 当前受支持的 Chrome、Edge、Firefox 或 Safari |

Python 3.13、Node.js 22 和 Caddy 已由容器镜像提供，Linux 宿主机不需要单独安装 Python、Node.js、MySQL 或 Web 服务器。

## 3. 部署前准备

1. 准备一个业务域名，例如 `crm.example.com`。
2. 在 DNS 服务商处添加 `A` 记录，指向 Linux 服务器公网 IPv4；如使用 IPv6，再添加正确的 `AAAA` 记录。
3. 云安全组和服务器防火墙允许入站 TCP `80/443`、UDP `443`，并保留 SSH 管理端口。
4. 确认运营商或云平台没有拦截 80/443，且域名已按所在地法规完成必要备案。
5. 将项目代码放到服务器，例如 `/opt/ConsultingRequirementManager`。

Caddy 需要从公网完成域名校验并自动申请 HTTPS 证书。DNS 未生效、80/443 不通或域名指向错误时，HTTPS 不会签发成功。

## 4. 安装 Docker

以下命令适用于 Ubuntu 24.04 LTS，使用 Docker 官方软件源：

```bash
sudo apt update
sudo apt install -y ca-certificates curl
sudo install -m 0755 -d /etc/apt/keyrings
sudo curl -fsSL https://download.docker.com/linux/ubuntu/gpg \
  -o /etc/apt/keyrings/docker.asc
sudo chmod a+r /etc/apt/keyrings/docker.asc

. /etc/os-release
echo "deb [arch=$(dpkg --print-architecture) signed-by=/etc/apt/keyrings/docker.asc] https://download.docker.com/linux/ubuntu ${VERSION_CODENAME} stable" \
  | sudo tee /etc/apt/sources.list.d/docker.list >/dev/null

sudo apt update
sudo apt install -y docker-ce docker-ce-cli containerd.io \
  docker-buildx-plugin docker-compose-plugin
sudo systemctl enable --now docker

sudo docker version
sudo docker compose version
```

验收时确认 Docker Engine 不低于 27.0，Compose 为 `v2.29` 或更高。其他 Linux 发行版应按 Docker 官方文档安装等价版本，不要使用来源不明的一键脚本。

## 5. 配置生产参数

进入远程版目录：

```bash
cd /opt/ConsultingRequirementManager/远程
cp .env.example .env
chmod 600 .env
```

生成适合写入数据库 URL 的十六进制随机密码：

```bash
openssl rand -hex 32
openssl rand -hex 32
openssl rand -hex 24
```

编辑 `.env`：

```bash
nano .env
```

必须修改以下值：

```dotenv
DOMAIN=crm.example.com
ACME_EMAIL=admin@example.com

MYSQL_DATABASE=consulting_requirement
MYSQL_USER=crm_app
MYSQL_PASSWORD=第一条随机值
MYSQL_ROOT_PASSWORD=第二条随机值

ADMIN_USERNAME=admin
ADMIN_PASSWORD=第三条随机值并补充大小写字母和特殊字符
```

要求：

- `DOMAIN` 只写域名，不加 `http://`、`https://` 或路径。
- MySQL 应用密码和 root 密码必须不同。
- 初始管理员密码至少 10 位，并包含大写字母、小写字母、数字和特殊字符。
- `ADMIN_USERNAME` 仅用于空用户库首次初始化，初始化成功后不得修改；修改后服务会拒绝启动，而不会静默创建另一个管理员。
- `.env` 不得提交到 Git、发送到聊天群或放入 Web 可访问目录。
- 同一套 MySQL 密码一经首次启动写入数据卷，后续不能只修改 `.env`；修改数据库密码必须同时在 MySQL 内执行正式轮换。

## 6. 启动服务

在 `远程` 目录执行：

```bash
sudo docker compose config
sudo docker compose pull
sudo docker compose build --pull
sudo docker compose up -d
sudo docker compose ps
```

正常状态应为：

- `mysql`：healthy
- `api`：healthy
- `web`：running

API 容器启动时会先执行 Alembic 数据库迁移，再校验/初始化种子数据，最后启动 FastAPI。迁移或首次管理员校验失败时 API 不会进入 healthy 状态。日志中应能看到当前迁移版本：

```bash
sudo docker compose logs --tail=200 api
sudo docker compose exec api alembic current
```

当前数据库迁移版本应为 `20260717_0006 (head)`。该结构包含 `artifact_change_uploads` 表，用于记录冻结版本成果物附件的暂存元数据、文件大小、SHA-256 摘要、变更申请、基线序号和上传人关联；真实暂存文件位于 `upload_data` 卷，不以二进制内容写入 MySQL。执行变更前系统会同时校验暂存文件是否存在、大小及 SHA-256，校验不一致时拒绝执行。

全新空库首次迁移需要创建全部表、索引和外键；在机械盘或资源较低的服务器上可能持续数分钟。只要 `api` 日志仍在输出 Alembic DDL 且 MySQL 保持 healthy，应等待迁移完成，不要因容器暂未 healthy 而中断进程。日常幂等重启不会重复执行已完成的迁移；生产部署和自动化验收建议为首次迁移预留至少 5 分钟。

API 的 MySQL 连接固定使用 `READ COMMITTED` 事务隔离级别，并配合项目、年度、版本和需求行锁处理并发预算写入。该设置由程序连接参数自动完成，不需要修改 MySQL 全局隔离级别；更新 API 镜像后必须重建并重启 `api` 容器，旧进程不会自动采用新的连接隔离配置。

查看启动日志：

```bash
sudo docker compose logs --tail=200 mysql api web
```

Compose 已为 MySQL、API 和 Web 配置 `json-file` 日志轮转：单文件最大 `10 MB`，每项服务最多保留 `5` 个文件，避免容器标准输出长期占满系统盘。

首次构建需要下载 MySQL 8.4、Python 3.13、Node.js 22 和 Caddy 镜像，耗时取决于服务器网络。

## 7. 防火墙与 HTTPS 验收

使用 UFW 时可执行：

```bash
sudo ufw allow OpenSSH
sudo ufw allow 80/tcp
sudo ufw allow 443/tcp
sudo ufw allow 443/udp
sudo ufw enable
sudo ufw status
```

不要执行 `ufw allow 3306` 或 `ufw allow 8000`。

DNS 生效后验证：

```bash
curl -fsS https://crm.example.com/health
curl -I https://crm.example.com/
```

健康检查应返回类似：

```json
{"status":"ok","database":"ok","timestamp":"2026-07-17T00:00:00"}
```

然后在外部电脑浏览器打开：

`https://crm.example.com/`

浏览器地址栏必须显示有效 HTTPS 证书，不能使用 `http://服务器IP:端口` 作为正式访问方式。

## 8. 首次登录与账号初始化

首次管理员只会在 `users` 表为空时创建。系统会校验管理员用户名格式和临时密码强度；已有用户数据但 `ADMIN_USERNAME` 与初始管理员不匹配时，服务拒绝启动，防止配置拼写错误意外创建高权限账号。

1. 使用 `.env` 中的 `ADMIN_USERNAME` 和 `ADMIN_PASSWORD` 登录。
2. 系统强制修改初始密码，完成前不能进入业务页面。
3. 进入“用户与权限”，立即创建第二个管理员账号。
4. 按实际人员预建咨询负责人、客户、销售、项目经理、研发人员和运营人员账号。
5. 为客户账号选择可访问项目；内部角色按单组织模式使用。
6. 管理员可按角色调整仪表盘组件的显示与顺序；布局保存在数据库中，同角色用户再次登录后继续使用该布局。
7. 用普通账号验证无权页面和资金/工时敏感字段不可见。客户只能查看本人需求，以及与本人需求关联且已审批的成果物；项目、年度和版本级材料默认不向客户公开。

系统不开放自助注册。普通用户忘记密码由管理员重置；管理员忘记密码由另一管理员重置。同一账号后登录会立即替换旧会话。

确认首个管理员已改密并已建立第二管理员后，可把 `.env` 改为 `ADMIN_PASSWORD=` 并重启；已有管理员密码不会被环境变量重置。必须保留 `ADMIN_PASSWORD` 变量名、不得修改 `ADMIN_USERNAME`，并继续以 `chmod 600` 保护 `.env`。

## 9. 数据持久化位置

Docker Compose 使用四个命名卷：

| 卷 | 内容 |
| --- | --- |
| `mysql_data` | MySQL 业务库 |
| `upload_data` | 正式成果物附件和冻结版本变更的待审批暂存附件 |
| `caddy_data` | HTTPS 证书和 Caddy 状态 |
| `caddy_config` | Caddy 运行配置 |

执行普通的 `docker compose down` 不会删除这些数据。禁止在生产环境执行：

```bash
docker compose down -v
```

`-v` 会删除命名卷并造成业务数据丢失。

数据库中的 `artifact_change_uploads` 暂存记录与 `upload_data` 中的暂存文件共同组成尚未执行的附件变更。备份、恢复和迁移时必须同时处理 MySQL 与 `upload_data`，不能只保留其中一项。

## 10. 备份与恢复

先创建仅管理员可读的备份目录：

```bash
cd /opt/ConsultingRequirementManager/远程
mkdir -p backups
chmod 700 backups
```

为保证数据库记录与附件引用一致，完整备份期间必须暂停 Web 和 API 写入，并为数据库与附件使用同一个备份编号：

```bash
(
  set -e
  BACKUP_ID="$(date +%F-%H%M%S)"
  trap 'sudo docker compose start api web >/dev/null' EXIT
  sudo docker compose stop web api
  sudo docker compose exec -T mysql sh -c \
    'exec mysqldump --single-transaction --routines --triggers -uroot -p"$MYSQL_ROOT_PASSWORD" "$MYSQL_DATABASE"' \
    > "backups/mysql-${BACKUP_ID}.sql"
  sudo docker run --rm \
    -v consulting-requirement-manager_upload_data:/data:ro \
    -v "$PWD/backups:/backup" \
    alpine:3.21 sh -c "tar -czf /backup/uploads-${BACKUP_ID}.tar.gz -C /data ."
  sha256sum "backups/mysql-${BACKUP_ID}.sql" "backups/uploads-${BACKUP_ID}.tar.gz" \
    > "backups/${BACKUP_ID}.sha256"
)
```

如果数据库或附件命令失败，不得把不完整文件标记为有效备份；先执行 `sudo docker compose start api web` 恢复服务，再排查原因。数据库 SQL、附件压缩包和校验文件必须使用同一个 `BACKUP_ID`。同时把 `.env` 保存到独立的加密密码库，不要和公开代码放在一起，并定期在隔离环境验证整套备份可恢复。

恢复会覆盖当前数据库和附件。操作前必须再次确认目标服务器、同一套 `BACKUP_ID` 和现有数据快照，然后停止业务写入：

```bash
(
  set -e
  BACKUP_ID="替换为需要恢复的备份编号"
  cd /opt/ConsultingRequirementManager/远程
  sha256sum -c "backups/${BACKUP_ID}.sha256"
  sudo docker compose stop web api
  sudo docker compose exec -T mysql sh -c \
    'exec mysql -uroot -p"$MYSQL_ROOT_PASSWORD" "$MYSQL_DATABASE"' \
    < "backups/mysql-${BACKUP_ID}.sql"
  sudo docker run --rm \
    -v consulting-requirement-manager_upload_data:/data \
    -v "$PWD/backups:/backup:ro" \
    alpine:3.21 sh -c \
    "find /data -mindepth 1 -maxdepth 1 -exec rm -rf -- {} + && tar -xzf /backup/uploads-${BACKUP_ID}.tar.gz -C /data"
  sudo docker compose start api web
  sudo docker compose ps
  curl -fsS https://crm.example.com/health
)
```

恢复后用管理员账号抽查项目、需求、成果物下载和审计记录。任何一步失败都应保持 Web/API 停止，使用恢复前快照回滚，不能让数据库与附件处于不同备份时间点后继续提供服务。

## 11. 更新程序

更新前完成数据库、附件和 `.env` 备份，并阅读当前业务说明。然后执行：

```bash
cd /opt/ConsultingRequirementManager/远程
sudo docker compose build --pull
sudo docker compose up -d
sudo docker compose ps
sudo docker compose logs --tail=200 api web
sudo docker compose exec api alembic current
curl -fsS https://crm.example.com/health
```

API 新容器会在提供服务前自动执行 Alembic 向前迁移。只部署已经在 MySQL 8.4 上通过迁移与业务测试的目标提交；遇到 `alembic_version` 缺失、数据库结构校验失败或迁移失败时必须停止上线并使用匹配备份恢复，不能手工标记迁移版本后强行启动。

如代码通过 Git 管理，应在备份后、构建前拉取已经验收的目标提交。不要直接部署未测试分支。数据库结构升级后不能只回退容器镜像；需要同时使用与该版本匹配的数据库备份恢复，禁止直接执行生产降级迁移。

## 12. 常见故障

### 域名打不开或证书签发失败

```bash
dig +short crm.example.com
sudo docker compose logs --tail=200 web
sudo ss -lntup | grep -E ':80|:443'
```

确认 DNS 指向当前服务器，云安全组和防火墙允许 80/443，且没有其他程序占用端口。

### 页面能打开但登录或数据请求失败

```bash
sudo docker compose ps
sudo docker compose logs --tail=200 api mysql
curl -fsS https://crm.example.com/health
```

健康检查显示 `database:error` 时，重点核对 MySQL 容器状态和 `.env` 密码。不要把数据库端口临时开放到公网排障。

`/health` 在数据库异常时返回 HTTP `503`，因此 API 容器也会退出 healthy 状态，避免 Caddy 把故障服务误判为正常。

### 修改 `.env` 后没有生效

```bash
sudo docker compose up -d --force-recreate api web
```

MySQL 已初始化后的用户密码不会因修改 `.env` 自动改变，必须执行数据库密码轮换流程。

### 同一账号在旧浏览器被退出

这是单账号单有效会话规则。新位置登录成功后，旧会话下一次请求会收到失效提示并回到登录页。多人使用时应给每个人建立独立账号，禁止共享管理员账号。

## 13. 上线验收清单

- `https://实际域名/` 可从用户网络访问，证书有效且自动跳转 HTTPS。
- 公网只开放必要的 SSH、80 和 443，`3306/8000` 不开放。
- `docker compose ps` 中 MySQL 和 API 为 healthy，Web 为 running。
- `docker compose exec api alembic current` 显示 `20260717_0006 (head)`，数据库存在 `artifact_change_uploads` 表及 `sha256_hex` 字段。
- `/health` 返回 `status: ok` 和 `database: ok`。
- 首个管理员已强制改密，第二管理员已建立。
- 客户项目白名单、本人需求范围、已审批成果物范围、角色页面权限和资金/工时字段隔离符合业务说明。
- 管理员调整任一角色的仪表盘组件顺序后，该角色重新登录仍使用已保存布局。
- 后登录能使同账号旧会话立即失效。
- 项目、年度、两个版本、需求、资金、成果物和运营记录可完成一轮业务流程。
- 跨版本比对在不足两个版本时给出正确提示，创建第二版本后可正常比对。
- 冻结版本成果物的新增/替换附件先进入暂存区并自动创建变更申请；上传人以外的有权人员批准、执行后正式附件和新基线同步生效，驳回或取消后暂存文件被清理。
- 同一基线上的重叠变更不能同时批准；任一变更执行并生成新基线后，旧基线申请不能继续审批或执行。
- MySQL 和附件备份已生成，并在隔离环境完成恢复验证。
