# MySQL 8.x LTS 远程版最终需求与部署验收文档

适用版本：应用 v1.6.3；数据库 MySQL 8.x（推荐 8.4 LTS）
基线日期：2026-07-16

完整业务需求和功能边界见仓库根目录 `最终需求与部署说明.md`。本文聚焦 Linux MySQL、多客户端、账号和附件存储的部署验收。

## 1. 架构定位

远程版是 Tkinter 桌面客户端。Linux 服务器运行 MySQL 8.x；客户端通过内网或 VPN 直接连接 MySQL。附件可统一选择服务器共享目录或阿里云 OSS，数据库保存附件定位信息。

此架构适合可信内网，不应把 MySQL 3306 暴露公网。公网、多租户或跨组织场景应增加 API 服务，避免桌面客户端持有数据库凭据。

## 2. 生产配置基线

```json
{
  "host": "10.0.0.10",
  "port": 3306,
  "user": "crm_user",
  "password": "",
  "password_env": "CRM_DB_PASSWORD",
  "database": "consulting_requirement_manager",
  "create_database": false,
  "seed_demo_data": false,
  "connect_timeout": 10,
  "read_timeout": 30,
  "write_timeout": 30,
  "ssl_ca": "C:/secure/mysql-ca.pem",
  "attachment_storage": "oss",
  "attachments_dir": "",
  "oss_endpoint": "https://oss-cn-hangzhou.aliyuncs.com",
  "oss_bucket": "your-private-bucket",
  "oss_prefix": "consulting-requirement-manager"
}
```

数据库密码通过 `CRM_DB_PASSWORD` 提供。首次管理员密码通过 `CRM_INITIAL_ADMIN_PASSWORD` 初始化，成功登录后应移除该变量。OSS 使用 `OSS_ACCESS_KEY_ID`、`OSS_ACCESS_KEY_SECRET`，临时凭据再设置 `OSS_SECURITY_TOKEN`。

正式验收设置 `CRM_REQUIRE_TLS=1` 和 `CRM_REQUIRE_STRICT_SQL=1`。健康检查会确认 MySQL 8.x、`utf8mb4`、严格 SQL mode、会话 TLS cipher、客户端目录以及附件后端。

## 3. 附件后端

### 服务器目录

`attachment_storage=server` 时，`attachments_dir` 必须是运行客户端可直接读写的路径。

- GUI 同时运行在 Linux 桌面：可使用 `/srv/consulting-requirement-manager/attachments`。
- GUI 运行在 Windows、附件位于 Linux：通过 SMB/NFS 挂载后使用 UNC 或映射路径。
- 多客户端必须访问同一份共享目录；每台电脑各自的 `data/attachments` 不构成共享存储。
- 使用专用两级子目录，不能指向磁盘根、共享根、用户主目录或应用/数据/配置目录；新附件以相对对象键入库，允许不同客户端挂载点不同。

目录账号只授予应用所需读写权限。应用内 ZIP 只备份附件，不备份 MySQL。

### 阿里云 OSS

`attachment_storage=oss` 时，Endpoint 必须为 HTTPS，Bucket 必须为私有，使用受限 RAM 用户或 STS 临时凭据。应用只允许操作配置的 `oss_prefix`，环境凭据轮换后会重建连接。启用版本控制、服务端加密、访问日志和生命周期；应用不批量下载 Bucket 做 ZIP 备份，恢复通过 OSS 对象版本完成。

## 4. 账号和权限

- 密码使用随机盐 PBKDF2-HMAC-SHA256 哈希保存。
- 不开放自助注册；管理员预建账号、分配或调整角色、启停用户、重置密码并可强制下线。
- 同一账号仅保留最后一次登录会话；后登录客户端替换前会话，旧客户端最长 10 秒内退出。
- 普通用户忘记密码由管理员重置；管理员由另一管理员重置，正式环境至少保留两个启用管理员。
- 新建客户账号默认无项目数据权限；管理员在顶部选择项目后，通过“授权当前项目/撤销当前项目”维护客户项目白名单。
- 角色由登录账号决定，远程版无可伪造的角色下拉框。
- 客户、研发人员、运营人员不显示资金页、资金详情和敏感导出。
- 客户和销售不显示研发预估工时、实际工时或工时明细。
- 用户管理和附件恢复仅管理员执行；冻结变更审批允许管理员和咨询负责人执行。
- 客户的项目下拉、项目列表、首页项目数和搜索结果按白名单过滤；内部角色按单组织模型查看全部项目。
- 页面进入、定时巡检和关键写操作重新校验账号状态、会话、角色和客户项目授权，账号停用、角色调整或项目授权撤销无需等待客户端重启。
- 操作权限由应用控制。由于客户端直连 MySQL，这不是数据库级零信任隔离。

## 5. 并发与一致性

- MySQL 连接启用自动提交，读取前执行连接存活检查和自动重连。
- 预算登记使用事务、`SELECT ... FOR UPDATE` 和版本汇总校验。
- 版本冻结、基线头、完整需求快照、已审批版本/需求成果物快照、冻结状态和日志在同一事务提交。
- 需求新增、编辑、分配和删除先锁定版本行，避免与冻结操作交错导致基线遗漏。
- 变更申请头、payload 和操作日志在同一事务提交；缺少 payload 的申请不能审批通过。
- 变更审批、需求更新/软删除、状态历史、审批状态和日志在同一事务提交。
- 冻结版本新增成果物先进入待审批状态；审批前重新校验成果物类型、挂载层级、项目归属和可见范围。
- 状态流转使用原状态条件更新，避免静默覆盖其他客户端已更新的状态。
- 服务器目录附件上传、删除、备份和恢复共用数据库命名锁，避免多客户端在文件复制或目录替换期间交错写入；恢复同名覆盖并保留备份中不存在的现有附件。

## 6. 备份

- MySQL：使用 `mysqldump --single-transaction`，关键环境同时开启 binlog 和异机备份。
- 服务器附件：文件系统快照或增量备份；应用 ZIP 可作为辅助，不替代服务端备份。
- OSS：Bucket 版本控制、跨区域复制和生命周期。
- 配置：单独备份不含明文密码的模板、CA 证书和运维参数。

建议每日全量备份，关键项目结合 binlog 达到 1 小时以内 RPO。季度执行数据库和附件联合恢复演练。

## 7. 日志与监控

每台客户端在 `CRM_DATA_DIR/logs` 输出 `runtime.log`、`error.log` 和 JSON Lines 格式的 `audit.log`，默认单文件 10MB、保留 30 份。使用 `CRM_LOG_LEVEL`、`CRM_LOG_MAX_BYTES`、`CRM_LOG_BACKUP_COUNT` 调整。MySQL `operation_logs.event_id/result` 可与文件审计对账；登录失败和权限拒绝同时写中央操作日志。

远程版客户端日志不会自动写到 Linux MySQL 服务器。正式多客户端环境必须通过 Filebeat、Fluent Bit、Vector 或企业代理集中采集，并同时监控 MySQL error log、slow query log、binlog、连接数、磁盘和备份。完整规则见仓库根目录 `日志与运维说明.md`。

## 8. 验收顺序

1. 执行远程离线检查：`python selftest.py`，默认不会连接 MySQL。
2. 在测试 MySQL 初始化表结构，确认无演示数据。
3. 初始化至少两个管理员，验证登录、本人改密、账号预建、角色调整、普通用户忘记密码重置、停用和强制下线。
4. 按七类角色逐一验证页面、字段、按钮和导出权限。
5. 完成项目到运维的业务主链路。
6. 使用两个客户端登录同一账号，确认后登录替换前会话且旧客户端退出不影响新会话；再并发登记预算、冻结同一版本和审批同一申请。
7. 验证服务器目录或 OSS 上传、异常回收、权限拒绝和备份恢复。
8. 仅在测试库同时设置 `CRM_MYSQL_INTEGRATION_SELFTEST=1` 和 `CRM_INTEGRATION_TEST_DATABASE=<测试库名>` 后执行连接集成检查：`python selftest.py`。配置中的库名必须完全匹配且包含 `test` 或 `selftest`。
9. 设置 `CRM_REQUIRE_TLS=1`，关闭同终端 GUI 后执行 `python healthcheck.py`，确认真实 MySQL、TLS、日志目录和服务器目录/OSS 写删探测全部为 `status=ok`。
10. 使用备份在独立测试库完成 MySQL 与附件联合恢复。

## 9. 上线门槛

必须同时满足：TLS 与网络白名单生效、数据库非 root、生产开关关闭演示数据、管理员密码已初始化、附件后端已验收、并发测试通过、备份恢复演练通过、客户端日志集中采集生效、MySQL 日志与监控生效、告警责任人明确。

当前尚无正式 schema 迁移框架和全面数据库外键/检查约束。1.6.3-mysql 首次连接旧库会在 MySQL 命名锁保护下补充审计、业务标识、工时、父需求、项目阶段、成果物审批、完整基线和用户会话等兼容字段和索引，需要一次性 `ALTER` 权限；测试库升级验证完成后可撤销。每次升级前必须先备份并在测试库验证；长期正式运维应补充版本化 SQL 迁移。

账号密码重置会下线原会话并提醒用户立即改密，但当前未强制首次改密，也未提供登录失败锁定、MFA 或应用内唯一管理员恢复。至少两名启用管理员和临时密码安全传递属于上线必须执行的运维控制；面向公网、跨组织或高安全等级场景，应先增加服务端 API、统一身份认证和登录风控。
