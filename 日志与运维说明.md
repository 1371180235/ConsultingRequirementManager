# 咨询项目全流程需求管理系统日志与运维说明

适用版本：SQLite 本地应用 v1.4.2、远程应用 v1.6.3（MySQL 8.x LTS）
基线日期：2026-07-16

## 1. 日志分层

| 日志 | 默认位置 | 内容 | 格式 |
| --- | --- | --- | --- |
| 运行日志 | `data/logs/runtime.log` | 启动、停止、数据库连接、权限拒绝、业务事务拒绝和运行状态 | UTF-8 文本 |
| 错误日志 | `data/logs/error.log` | 数据库错误、未捕获 Tkinter 回调异常、启动致命错误和完整堆栈 | UTF-8 文本 |
| 文件审计日志 | `data/logs/audit.log` | 登录、权限拒绝、业务新增/修改/审批/状态/资金等审计事件 | JSON Lines |
| 数据库审计日志 | SQLite/MySQL `operation_logs` | 可在系统设置查询的业务操作记录、前后值和描述 | 数据库表 |
| MySQL 服务端日志 | Linux MySQL 日志目录 | MySQL 错误、慢查询、连接和服务状态 | 由 MySQL 管理 |

运行日志包含时间、级别、进程 ID、客户端主机名和消息。审计 JSON 包含 `timestamp`、`event_id`、`host`、`process_id`、`app_version`、`variant`、`operator`、`object_type`、`object_id`、`operation_type`、`description` 和 `result`。`event_id` 使用 UUID4，数据库以唯一索引约束。数据库 `operation_logs` 同步保存 `event_id` 与 `result`，可与文件审计逐条核对；历史记录升级后事件 ID 为空且结果标记为 `legacy`，不会误标为成功。

应用不会把登录密码、数据库密码、OSS AccessKey、SecurityToken 或 SQL 参数写入文件日志。数据库错误只记录 SQL 结构摘要，不记录参数值。

系统设置支持按对象类型和关键词筛选最近 200 条数据库审计记录，关键词覆盖操作者、操作、结果、事件 ID 和说明。对象类型包含项目/年度/版本/需求、资金流水、资金申报、运营服务、成果物、变更申请、用户授权、备份和健康检查。管理员/咨询负责人可导出全部匹配记录为 CSV；导出应用公式注入防护。远程版登录失败与权限拒绝同时进入 MySQL 中央操作日志和客户端文件审计；中央写入失败时仍保留本地失败审计和错误日志。

项目阶段、资金申报状态、运营服务记录、父需求变更、成果物审批和预算修改都应记录前后值。冻结版本时，基线头、完整需求快照、已审批成果物快照、冻结状态和审计事件必须在同一数据库事务中提交。

## 2. 轮转与保留

三类文件日志都按大小轮转。默认单文件最大 10MB，每类保留 30 个历史文件：

```text
runtime.log
runtime.log.1
runtime.log.2
...
```

可通过环境变量调整：

```powershell
$env:CRM_LOG_LEVEL = "INFO"
$env:CRM_LOG_MAX_BYTES = "10485760"
$env:CRM_LOG_BACKUP_COUNT = "30"
```

范围限制：单文件 1MB-1GB，历史份数 1-365。生产环境建议 `INFO`；临时排障可改为 `DEBUG`，排障结束后恢复，避免日志增长过快。

不要再对应用文件日志叠加会重命名同一文件的系统 `logrotate`。如需长期留存，应由 Filebeat、Fluent Bit、Vector 或企业日志代理读取当前文件和轮转文件，发送到集中日志平台。

标准 `RotatingFileHandler` 不提供多进程轮转锁。同一终端不要让多个 GUI 实例共用同一个 `CRM_DATA_DIR`，也不要在 GUI 运行时并发执行 `healthcheck.py`。多终端必须使用各自的本地数据/日志目录，再由采集代理集中汇总。

数据库 `operation_logs` 默认不自动删除。应根据合同、审计和数据安全制度确定归档周期，先导出到只读存储，再由 DBA 执行清理。建议普通项目保留不少于 1 年，涉及验收、采购或资金审计的项目按业务制度延长。

## 3. 运行目录

默认运行数据位于程序目录下 `data/`。正式安装到不可写目录时必须指定可写数据目录：

```powershell
$env:CRM_DATA_DIR = "D:\ConsultingRequirementManagerData"
```

MySQL 远程版还可独立指定配置目录：

```powershell
$env:CRM_CONFIG_DIR = "D:\ConsultingRequirementManagerConfig"
```

禁止把以上目录配置为磁盘根目录。Windows 应使用仅授权业务用户和运维账号访问的 NTFS 目录；Linux 日志目录会尝试设置为 `0700`，日志文件设置为 `0600`。

远程版日志产生在每台桌面客户端，不在 MySQL 服务器集中生成。多客户端正式使用时必须部署日志采集代理，并以 `host`、`operator` 和 `event_id` 区分来源。

覆盖升级时安装器只原子替换主程序与启动脚本，保留现有 `data/`；远程版还保留 `config/mysql_config.json`，仅在缺失时安装示例配置。安装后默认不自动启动，便于运维先核对目录权限、配置和 TLS 证书。升级后应先关闭旧客户端，再执行健康检查，避免多个进程同时轮转同一日志。

## 4. 关键事件

重点监控以下运行日志关键字：

| 关键字 | 含义 | 建议级别 |
| --- | --- | --- |
| `application_fatal_error` | 应用启动或主循环致命错误 | 立即告警 |
| `unhandled_tk_callback` | 未被业务代码处理的界面操作异常 | 立即告警 |
| `mysql_connection_unavailable` | MySQL 断连且重连失败 | 立即告警 |
| `transaction_failed` | 预算、冻结、需求/成果物审批、资金申报或状态事务发生系统错误 | 立即告警 |
| `sqlite_execute_failed` / `mysql_execute_failed` | 数据库写失败 | 立即告警 |
| `login_failed` | 登录失败 | 5 分钟内集中出现时告警 |
| `permission_denied` | 角色尝试执行无权限操作 | 异常增长时审查 |
| `transaction_rejected` | 业务规则正常拒绝，如超预算或版本冻结 | 统计观察，不作为系统故障 |

建议同时监控：数据盘剩余空间低于 20%、日志采集延迟超过 5 分钟、备份连续失败、MySQL 连接数超过 80%、慢查询突增、OSS 上传错误率和客户端时间偏差。

## 5. 时间与审计

所有客户端和 Linux MySQL 服务器必须启用 NTP/域时间同步。审计时间目前使用客户端本地时间；时间漂移会影响跨客户端事件排序。

`operation_logs` 和 `audit.log` 是双重审计，但都不能等同于不可篡改审计仓。需要强合规时，应把 `audit.log` 实时发送到限制删除权限的集中平台或 WORM/对象锁定存储，并限制数据库管理员直接修改 `operation_logs`。

## 6. MySQL 服务端日志

应用日志不能替代 MySQL 服务端日志。Linux 正式环境至少启用：

- MySQL error log：服务启动、崩溃、存储和权限错误。
- slow query log：建议初始阈值 1-2 秒，稳定后按负载调整。
- binlog：用于时间点恢复，配置合理保留周期和异机备份。
- 操作系统日志：磁盘、内存、OOM、文件系统和网络异常。

不要在 MySQL general log 中长期记录全部 SQL，它会产生较大性能和敏感数据风险。仅在受控排障窗口短时开启。

## 7. 排障顺序

1. 查看界面提示的 `error.log` 路径。
2. 按时间检查同主机 `runtime.log` 上下文。
3. 用 `event_id`、操作者、对象和时间核对 `audit.log` 与 `operation_logs`。
4. 远程版同时检查 MySQL error/slow log、网络和 TLS。
5. 附件问题按存储后端检查共享目录权限或 OSS 请求/审计日志。
6. 修复后执行对应版本 `selftest.py`，不要直接在生产数据库执行开发性写入测试。
7. 远程 `selftest.py` 默认只执行离线检查；只有测试库才同时设置 `CRM_MYSQL_INTEGRATION_SELFTEST=1` 和 `CRM_INTEGRATION_TEST_DATABASE=<测试库名>` 执行连接集成测试。配置中的库名必须完全匹配，且包含 `test` 或 `selftest`。
8. 安装器变更后执行 `python -m unittest -v installer_selftest.py`，验证全新安装、覆盖升级、数据/配置保留和非法路径拒绝。
9. 关闭同终端 GUI 后执行 `healthcheck.py`，以退出码和 JSON 输出确认真实数据库、目录、日志、TLS 和附件后端健康；远程正式环境应确认返回非空 `tls_cipher`。

## 8. 上线验收

- [ ] 运行、错误、审计三类日志均生成，中文和 JSON 编码正常。
- [ ] 人工触发测试环境异常后，`error.log` 包含堆栈且界面只显示必要信息。
- [ ] 登录成功/失败、权限拒绝、预算、项目阶段、资金申报、运营服务、冻结、需求/成果物审批和状态流转可在审计中定位。
- [ ] 文件达到阈值后能轮转，旧文件份数符合配置。
- [ ] 日志目录权限符合最小授权，备份中不包含明文密钥。
- [ ] 多客户端日志已采集到集中平台，主机时间同步。
- [ ] MySQL error、slow、binlog 和备份告警已配置责任人。
- [ ] 日志归档、保留、脱敏、查询和销毁流程已经书面确认。
