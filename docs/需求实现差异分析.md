# 需求实现差异分析

分析日期：2026-06-23

代码范围：`app.py`、`说明文档.md`、`版本变更记录.md`

## 1. 当前代码实现概览

当前项目是一个单文件 Tkinter + SQLite 桌面端 MVP，具备基础数据管理能力。

已具备能力：

- 本地 SQLite 数据库初始化。
- 项目、年度计划、落地版本、需求任务、资金流水、成果物、用户、日志、变更申请表结构。
- 示例数据初始化。
- 左侧导航 + 顶部项目/年度/版本/角色选择器。
- 首页指标卡和最近需求列表。
- 项目、年度计划、版本、需求、资金、成果物基础新增和列表展示。
- 需求状态流转和操作日志记录。
- 按角色隐藏部分敏感资金字段。
- 全局需求关键词搜索。
- 需求、资金、成果物 CSV 导出。
- 数据库和附件 ZIP 备份恢复。
- 版本冻结字段和冻结操作。

## 2. 需求与现状对照表

| 需求模块 | 目标能力 | 当前实现 | 差距等级 | 说明 |
| --- | --- | --- | --- | --- |
| 多角色工作台 | 不同角色展示专属指标、列表和操作 | 仅通过右上角角色选择展示不同提示，需求表对敏感角色隐藏预算字段 | 高 | 未实现菜单权限、数据行权限、看板组件配置和角色专属操作 |
| 全局搜索 | 秒级定位需求所在项目、版本、状态和资金 | 支持需求编号、名称、描述、标签搜索 | 中 | 搜索结果未展示项目、年度、版本、预算、负责人，也不能直达详情 |
| 需求统一收口池 | 所有需求进入统一需求池，未定版本进入待规划池 | 新建需求直接落入当前版本；`version_id` 可为空但 UI 不支持待规划池 | 高 | 需要新增待规划池视图和需求分配版本动作 |
| 四维关联模型 | 需求关联规划、版本、角色、资金 | 需求已关联项目、年度、版本、来源角色和预算字段 | 低 | 基础模型已有，需补充 UI 展示和校验 |
| 多标签分类 | 标签可自定义、多选、筛选 | 标签是普通文本 | 中 | 缺少标签字典、多选控件和标签筛选 |
| 状态机流转 | 按流程状态流转并记录日志 | 支持手动选择任意状态并记录日志 | 中 | 缺少状态流转规则、权限限制和异常状态处理逻辑 |
| 版本隔离 | 切换版本时列表、图表、成果物联动 | 需求按当前版本过滤；资金指标按当前版本统计 | 中 | 成果物没有按当前版本过滤；资金流水按项目展示；缺少完整联动 |
| 版本基线 | 发布/冻结后锁定核心字段 | 有 `is_frozen` 和冻结按钮 | 高 | 冻结后仍可新增/变更需求和成果物，变更申请表未接入流程 |
| 跨版本比对 | 需求、预算、成果物差异对比 | 未实现 | 高 | 需要新增比对页面和差异计算 |
| 资金四级穿透 | 项目 -> 年度 -> 版本 -> 需求流向图 | 资金页展示 5 个指标卡和流水表 | 高 | 无图形化拓扑/桑基/树状图，流水未关联需求 UI |
| 预算预警 | 超预算红色、剩余绿色、无预算灰色 | 未实现 | 中 | 需要预算计算和样式高亮 |
| 成果物管理 | 按业务阶段和挂载对象结构化管理 | 支持上传附件并记录挂载对象 | 中 | 列表未按当前上下文过滤，缺少阶段时间轴和上传提醒 |
| 阶段里程碑 | 展示 6 个业务阶段和当前阶段 | 未实现 | 高 | 需要项目流程页或首页流程组件 |
| 验收触发器 | 版本需求全部完成后提醒上传验收报告 | 未实现 | 中 | 需要检测版本需求状态并展示提醒 |
| 报表导出 | 按角色导出定制报告 | 仅导出原始 CSV | 中 | 需要角色报表模板和过滤逻辑 |
| 权限模型 | 菜单级、字段级、行级 RBAC | 仅模拟角色切换和字段隐藏 | 高 | 需要用户登录、权限表、查询过滤 |
| 操作日志 | 关键操作全记录 | 已记录新增、状态变更、冻结、上传、备份 | 中 | 预算字段修改、恢复备份、变更审批等未覆盖 |
| UI 美化 | 企业级 SaaS，图表、卡片、趋势和高亮 | 基础 ttk 桌面界面 | 高 | 缺少统一视觉体系、图表、筛选、详情面板和高亮 |

## 3. 数据模型差距

当前表结构已覆盖主要实体，但存在以下不足：

- 缺少权限相关表：角色权限、菜单权限、字段权限、数据范围权限。
- 缺少标签字典表：目前标签是逗号文本，无法统一管理。
- 缺少需求状态流转记录表：操作日志可追溯，但不便于专门统计状态流转。
- 缺少版本基线快照表：冻结版本后无法保存需求和成果物快照。
- 缺少看板配置表：无法配置不同角色的工作台组件。
- 资金流水有 `requirement_id` 字段，但新增资金流水 UI 未提供需求选择，实际无法穿透到单需求。
- 成果物有 `version_no` 字段，但新增成果物 UI 未填写版本号。

## 4. UI 与交互差距

- 首页只有指标卡和需求表，尚未体现角色化看板。
- 表格不支持筛选、排序、详情查看、编辑、删除和批量操作。
- 新增表单较朴素，缺少必填校验、数字校验、日期校验。
- 顶部上下文选择已经具备，但成果物和资金流水未完全跟随上下文过滤。
- 资金页没有图形化视图。
- 版本页没有比对、基线详情和变更入口。
- 需求页没有待规划池、详情面板和按标签/来源/状态快速筛选。

## 5. 推荐研发优先级

### 第一阶段：补齐闭环和一致性

1. 新增正式需求文档和差异分析文档。
2. 修复 `说明文档.md` 编码问题并补充运行说明。
3. 统一 UI 主题、按钮、表格、指标卡样式。
4. 让成果物、资金流水跟随当前项目/年度/版本过滤。
5. 为资金流水新增需求选择，使其真正关联单个需求。
6. 新增待规划池，支持将未分配版本的需求分配到版本。
7. 补充表单校验和错误提示。

### 第二阶段：实现核心需求增强

1. 资金四级穿透可视化。
2. 版本冻结后的只读限制和变更申请流程。
3. 跨版本需求/预算/成果物比对。
4. 阶段里程碑和成果物提醒。
5. 搜索结果补充项目、年度、版本、预算和负责人。

### 第三阶段：权限和报表

1. 用户登录和角色权限配置。
2. 菜单级、字段级、数据行级权限。
3. 按角色导出定制报告。
4. 管理员配置角色看板组件。

## 6. 下一步代码改造建议

建议先在当前 Tkinter 架构上做增量增强，避免一次性重构导致功能回退。

优先改造顺序：

1. 重构样式和通用表格/表单组件。
2. 增强首页看板、需求页、资金页、成果物页。
3. 增加待规划池、资金穿透和版本比对。
4. 再做权限、登录和变更流程。

若后续需要 Web 化，可将现有 SQLite 表结构迁移到 FastAPI + Vue/React，但当前需求也可以继续用桌面端实现。
