# 咨询项目全流程需求管理系统业务说明

更新日期：2026-07-17

本文是项目唯一的业务说明文档，描述本地版与远程版当前提供的业务能力、实际使用规则和交付边界。Windows 单机版的安装、升级、卸载和数据保存规则统一写在本文第 5 节；Linux 服务器版的环境要求、安装命令、HTTPS 访问地址、备份和升级步骤见 `远程/部署说明.md`。

## 1. 产品定位

系统面向咨询项目负责人，以资金规划为主轴、落地版本为隔离边界、需求任务为最小管理颗粒度，统一管理从宏观规划到上线运营的全过程。

统一业务链路：

`规划项目 -> 年度计划 -> 落地版本 -> 需求任务 -> 资金/成果物 -> 验收 -> 运维运营`

本地版和远程版使用同一套业务规则、页面能力和数据口径。两者只在运行方式、数据存储位置、身份认证和多人协作能力上不同：

- 本地版是 Windows 单人单机程序，数据保存在用户选择的安装路径。
- 远程版完整运行在 Linux 服务器，用户仅使用浏览器访问 `https://业务域名`。

## 2. 业务角色

| 角色 | 核心视图 | 主要业务操作 |
| --- | --- | --- |
| 管理员 | 全部业务、用户权限、审计 | 维护基础数据；远程版预建账号、分配角色、客户项目授权和密码重置 |
| 咨询负责人 | 项目规划、版本、需求、资金、审批 | 创建项目/年度/版本，统筹需求和资金，审批版本变更 |
| 客户 | 已授权项目、本人需求、版本进展 | 提交新需求、查看处理状态和版本规划；不查看资金成本和研发工时 |
| 销售 | 项目进展、资金申报、交付材料 | 提交需求、维护资金申报、上传成果物；不查看研发内部工时 |
| 项目经理 | 版本交付、验收、投入、风险 | 管理需求状态、版本投入、成果物和验收 |
| 研发人员 | 版本任务、优先级、工时 | 领取任务、更新研发状态、填报本人任务工时；不查看资金成本 |
| 运营人员 | 上线问题、推广、维护记录 | 登记线上问题和运营服务，关联原需求与上线版本；不查看资金成本 |

本地版不登录，右上角角色选择器用于单人预览不同角色的工作台和字段视角，不构成身份隔离。需要独立复核的冻结版本变更由操作人在审批时填写另一名实际复核人的姓名，系统阻止复核人与申请人同名并记录审批意见；该姓名用于单机流程留痕，不等同于远程账号认证。远程版由登录账号确定角色，权限由服务端强制校验，不能通过浏览器修改请求绕过。

## 3. 六阶段项目流程

| 阶段 | 核心工作 | 主要成果物 | 固定挂载层级 |
| --- | --- | --- | --- |
| 1. 宏观规划 | 明确项目方向和总体资金盘 | 可研报告 | 项目 |
| 2. 规划细化 | 拆分年度任务、痛点、功能和预算 | 分年任务申报书 | 年度计划 |
| 3. 建设落地 | 细化功能点和交付要求 | 任务书方案、需求任务表、任务清单 | 落地版本 |
| 4. 招投标 | 明确标的范围和招应标要求 | 招标文件、应标文件 | 落地版本 |
| 5. 项目交付验收 | 核对版本交付并归档总结 | 验收报告、项目总结 | 落地版本 |
| 6. 运维运营 | 处理上线问题、推广和维护 | 运维反馈、运营反馈 | 需求任务 |

流程里程碑页按六个阶段显示当前进度、成果物数量和验收缺口。当版本需求已经进入待验收、上线或关闭状态，但缺少已审批的验收报告/项目总结时，系统给出补充材料提醒。

## 4. 统一功能范围

### 4.1 多角色工作台

- 顶部统一选择当前项目、年度和版本，切换后页面数据联动刷新。
- 首页显示角色待办、项目指标、需求状态分布、推进趋势和最近需求。
- 支持专业蓝、清雅绿、暖灰橙、深色模式四套主题，一键切换并保存偏好。
- 无权限的页面、按钮和敏感字段不显示；远程版同时由 API 再次校验权限。

### 4.2 项目、年度与版本

- 项目保存项目编号、名称、总预算、当前阶段和说明，项目编号唯一。
- 年度计划归属一个项目，记录年度预算、业务痛点和年度目标。
- 落地版本归属一个年度计划，记录版本编号、目标、预算和状态。
- 项目内年度预算合计不能超过项目总预算；年度内版本预算合计不能超过年度预算。
- 远程版只有管理员和咨询负责人可以创建、修改项目、年度计划和落地版本，以及冻结、发布版本、审批和执行版本变更；删除项目、年度计划或草稿版本仅限管理员。项目经理、销售和其他角色可以按权限查看规划数据，但不能改写规划结构。
- 顶部项目切换联动年度，年度切换联动版本，避免跨项目、跨年度误挂数据。

### 4.3 版本隔离、基线与比对

- 每条需求、版本成果物、版本预算和基线均明确关联落地版本。
- 版本冻结时生成包含版本信息、需求、预算和成果物的基线快照；冻结后，版本信息以及需求的编号、标题、版本归属、预算等范围字段不能直接修改。需求仍可按交付流程领取、流转执行状态和填报工时，这些执行记录不改变已冻结的需求范围。
- 冻结版本的版本信息、需求或成果物调整必须提交结构化变更申请。“批准”和“执行变更”是两个明确步骤：批准只记录决策，不修改正式业务；执行时再次校验基线，更新业务数据、生成递增的新基线并记录审计。
- 远程版由申请人以外的有权账号审批，服务端按登录身份强制阻止自审。本地版没有账号身份隔离，审批时必须填写与申请人不同的实际复核人姓名和审批意见，系统据此阻止同名自审并留痕；执行人另行确认后才能使变更生效。
- 远程版每项变更申请记录提交时的基线序号。审批和执行都会核对当前基线；其他变更已经执行并生成新基线后，基于旧基线的申请不能继续审批或执行，必须按最新基线重新提交。
- 远程版会阻止同一基线上的已审批申请相互覆盖：同一版本字段、同一需求、同一成果物或同一新增成果物目标发生重叠时，后续申请不能批准；互不重叠的申请可以分别批准，但任一申请执行并生成新基线后，其余未执行申请即按基线过期处理。
- 跨版本比对至少需要同一项目存在两个落地版本；不足两个版本时页面说明原因，并提供创建第二个版本的入口。
- 两个待比对版本可以属于同一年度或不同年度。需求按稳定业务标识对齐，不按每个版本中可能不同的需求编号强行匹配；结果展示新增、移除、修改和一致项，并按当前角色权限展示预算、成本和工时差异。

### 4.4 需求全生命周期

- 所有角色提交的需求进入统一需求池，创建时必须关联规划项目。
- 版本未确定时进入待规划池，可暂不关联年度和版本；分配到未冻结的落地版本时，系统按目标版本补齐所属年度。
- 完成规划后的需求关联所属规划、落地版本、来源/对接角色和占用资金四个维度；待规划需求会明确显示尚未分配版本和资金。
- 标签来自统一标签字典并支持多选；支持业务痛点、功能优化、运维 Bug、招投标要求等分类。
- 可关联同项目的原需求；系统拒绝关联自身、跨项目关联和循环关联。
- 研发可领取已排期任务并填报预估/实际工时。

需求状态按受控状态机流转：草稿、规划中、已排期、研发中、待验收、已上线运维、已关闭、已驳回、已挂起、已取消、变更中、退回修改。每次状态变更必须填写说明，并与状态历史、操作日志在同一事务保存。

### 4.5 资金全链路

- 按“项目总预算 -> 年度预算 -> 版本预算 -> 需求资金”四级展示资金流向。
- 项目、年度和版本字段记录各级计划预算；需求资金流水支持预算分配、实际消耗和预算调整。除调整金额可正可负且不能为 0 外，其余流水金额必须大于 0；所有金额必须有限、层级一致，调整后需求分配预算不能小于 0。
- 资金树可逐级展开和下钻，高亮预算超限、执行偏差和未分配资金。
- 资金树不会遗漏尚未分配版本的需求：已选年度但未选版本的需求归入该年度下的“年度内待规划”分支，尚未选择年度和版本的需求归入项目下的“无年度待规划”分支。待规划分支用于核对未分配范围，需求分配到正式年度和版本后才能登记资金流水。
- 客户和研发不查看资金成本；销售不查看研发工时；管理员、咨询负责人和项目经理可按职责查看完整投入。
- 远程版资金流水只能由管理员、咨询负责人或项目经理登记，类型固定为预算分配、实际消耗和预算调整；销售可以查看资金并维护资金申报，但不能登记资金流水。
- 资金申报独立管理草稿、已提交、审批中、已批复、已驳回和已拨付状态，并校验项目、年度、版本归属及申报金额。
- 只有申请人可以编辑草稿或已驳回的申报，也只有申请人可以首次提交或在修改后重新提交；已进入审批流程的内容不能直接改写。
- 有审批职责的角色推进审核、批复、驳回和拨付。远程版禁止申请人审核、批复或拨付本人申报，销售账号只能创建、提交以及修改被驳回的申报。
- 新建、编辑、提交、重新提交、审批、驳回、批复和拨付均保留操作人、时间、前后状态等审计信息。

### 4.6 成果物、运营、搜索与报表

- 成果物按六阶段类型上传，并校验项目、年度、版本、需求的挂载层级。
- 新上传成果物先保存为草稿；草稿或已驳回成果物可提交审批，有权角色可审批通过或驳回。驳回后可重新提交，审批意见和全程状态变更均进入审计记录。
- 远程版禁止上传人审批自己的成果物。远程客户只能查看本人需求关联且审批通过的成果物；本地客户视角只显示审批通过并标记为“客户可见”的成果物。里程碑完成度和验收材料缺口只统计已审批成果物。
- 远程版对冻结版本范围内的成果物新增或附件替换使用真实文件送审：文件先写入服务器暂存区，系统同时创建绑定当前基线的变更申请；审批和执行前，正式成果物及版本基线均不改变，客户也不能查看暂存附件。
- 冻结成果物附件变更必须由上传人以外的有权人员审批，批准后还需执行才会转为正式附件并生成新基线。申请被驳回或在执行前取消时，暂存记录和暂存文件会被清理，正式成果物不受影响。
- 运营服务记录支持线上 Bug、问题解答、推广活动、维护记录和功能建议，可关联原需求及版本。
- 全局搜索可按需求编号、名称、描述、标签、项目、版本、成果物和运营内容查找。点击结果时系统先准确切换到该记录所属的项目、年度和版本，再定位具体项目、版本、需求、成果物或运营记录；未分配版本的需求会切换到对应项目的待规划池。
- 支持需求、资金、资金申报和成果物等 CSV 导出，导出内容按角色隐藏敏感字段，并防止表格公式注入。
- 关键业务新增、修改、删除、状态流转、预算调整、审批和权限拒绝均记录审计信息。

## 5. 本地版交付与使用

本地版最终交付文件为：

`本地/ConsultingRequirementManagerSetup.exe`

客户只需使用 Windows 10/11 64 位电脑，不需要安装 Python、SQLite、MySQL 或其他数据库。使用步骤如下：

1. 双击 `ConsultingRequirementManagerSetup.exe`。
2. 阅读《软件最终用户许可协议》，勾选“我已阅读并接受”后才能进入下一步；拒绝协议不会创建安装目录。
3. 自主选择一个可读写的安装路径，例如 `D:\ConsultingRequirementManager`，然后完成安装。
4. 安装后从安装目录启动主程序；程序首次运行自动创建本地数据库结构。

正式安装包首装不预置演示数据。程序首次启动后，17 张业务/审计表均为 0 行，项目、年度计划、落地版本、需求、资金、资金申报、成果物、运营记录、基线、变更申请和业务审计页面均从空数据开始；系统仅初始化 1 个可用的单机默认管理员记录和 8 个启用的基础标签，不生成虚假的初始化审计事件。默认标签为“业务痛点”“功能优化”“运维 Bug”“招投标要求”“验收整改”“客户新增”“版本必做”“待确认”；管理员可在标签字典中新增、启用或停用标签。交付给客户的 Setup 不携带研发和演示期间生成的 `app.db`。

默认目录结构：

```text
用户选择的安装目录/
  ConsultingRequirementManager.exe
  UninstallConsultingRequirementManager.exe
  启动咨询项目需求管理系统.bat
  EULA_zh-CN.txt
  .crm-install.json
  data/
    app.db
    attachments/
    backups/
    exports/
    logs/
```

数据库、附件、导出、备份和日志全部位于所选安装路径的 `data/`，不会写入远程服务器。关闭并重新打开程序不会清空数据。使用新版 Setup 覆盖安装到同一路径时，安装器更新程序文件并保留已有 `data`，程序启动时按当前结构兼容升级本地数据库。

可从 Windows“已安装的应用”或安装目录中的 `UninstallConsultingRequirementManager.exe` 卸载。卸载界面让用户选择保留业务数据或同时删除业务数据：默认保留 `data/`；勾选删除后还必须进行第二次不可恢复确认，取消任一次确认均不删除业务数据。重要数据仍应定期导出 ZIP 备份并验证可恢复。

本地版用于单人单机，不支持多人同时使用、跨电脑共享数据或正式账号权限隔离。

## 6. 远程版交付与使用

远程版的 Web 前端、服务端 API、MySQL 数据库和附件目录全部部署在 Linux 服务器。用户设备只需要主流浏览器，通过以下形式访问：

`https://业务域名`

浏览器只与 HTTPS Web/API 通信，不安装客户端，不持有 MySQL 地址、端口、账号或密码。MySQL 8.4 LTS 仅在服务器容器内部网络提供服务，不直接暴露到公网。

远程版首次初始化只创建首个管理员和默认标签，不预置项目、年度、版本、需求、资金或成果物等演示业务数据。远程版与本地版使用同一组默认标签：“业务痛点”“功能优化”“运维 Bug”“招投标要求”“验收整改”“客户新增”“版本必做”“待确认”。首个管理员登录并修改临时密码后，应在系统中创建第二名管理员和其他业务账号。

远程账号规则：

- 不开放自助注册，由管理员在“用户与权限”页面预建账号并分配一个固定角色。
- 客户账号由管理员分配可访问项目白名单；内部角色按单组织模式使用。
- 普通用户忘记密码由管理员重置；管理员忘记密码由另一名管理员重置。
- 新建或重置密码后，用户首次登录必须修改临时密码。
- 连续登录失败达到限制后暂时锁定账号，降低暴力尝试风险。
- 同一账号同一时间只保留一个有效会话；后登录成功后，旧浏览器会话在下一次请求时立即失效并回到登录页。
- 停用账号、修改角色或重置密码会使原会话失效。

正式环境建议至少保留两名启用的管理员。服务器环境、域名解析、HTTPS、备份和升级步骤以 `远程/部署说明.md` 为准。

## 7. 两版差异边界

| 维度 | Windows 本地版 | Linux 远程版 |
| --- | --- | --- |
| 业务功能 | 与远程版一致 | 与本地版一致，另提供真实用户与权限管理 |
| 使用方式 | 双击 Setup 安装后运行桌面程序 | 浏览器访问 `https://业务域名` |
| 使用人数 | 单人单机 | 同一组织多人协作 |
| 数据库 | 程序内置 SQLite，随安装路径保存 | 服务器内 Oracle MySQL 8.4 LTS |
| 数据库凭据 | 用户无需配置数据库 | 只保存在服务器，浏览器不可见 |
| 身份权限 | 不登录，角色仅用于视图预览 | 服务端登录、RBAC、客户项目白名单 |
| 附件 | 所选安装路径的 `data/attachments` | Linux 服务器持久化目录 |
| 备份 | 应用内 ZIP 备份数据库和附件 | 服务器分别备份 MySQL、附件和部署配置 |
| 网络 | 可完全离线 | 必须能访问部署域名 |

## 8. 演示验收主线

演示或验收时按以下主线操作，可覆盖核心需求：

1. 新建项目并设置总预算。
2. 新建年度计划并拆分年度预算。
3. 在同一项目下创建至少两个落地版本。
4. 新建待规划需求，选择多个标签，关联来源角色和原需求，再分配到版本。
5. 领取研发任务并按状态机推进到待验收/上线。
6. 分别创建年度内待规划和无年度待规划需求，核对资金树分支；将需求分配到版本后登记资金分配和实际消耗，再逐级穿透到需求。
7. 新建资金申报，完成草稿编辑、提交、审批；再模拟一次驳回、修改和重新提交，核对审计记录。
8. 上传六阶段成果物，从草稿提交审批，由有权角色审批通过；再模拟一次驳回后重新提交。
9. 冻结版本生成基线，提交版本变更和真实附件变更，由另一复核人批准后再单独执行，核对“批准前业务不变”和“执行后正式附件及新基线生效”；远程版再验证驳回或取消会清理暂存附件。
10. 在同一基线上提交重叠变更并验证系统阻止批准；执行一项变更后，验证其余旧基线申请不能继续审批或执行。
11. 按稳定业务标识对比两个版本，查看新增、移除、修改及预算/工时差异。
12. 登记运营问题并关联上线需求。
13. 使用全局搜索逐项定位需求、成果物和运营记录，再用 CSV 导出和审计日志核对完整链路。

本地版额外验收协议门禁、可选安装路径、覆盖升级保留数据和卸载数据选择；远程版额外验收账号预建、首次改密、角色权限、客户项目授权、单账号单会话和 HTTPS 域名访问。
